# RF SaaS â€” Protocol v0.5

Architecture: agent-driven live stream, with a serverâ†’agent control channel for
live config updates initiated by viewers.

**Diff from v0.4**:
- `protocol_version` bumped from `"0.3"` to `"0.5"` (v0.4 was a wire-encoding
  extension and kept the `"0.3"` string; v0.5 adds new message types and a new
  optional field, so the version string moves).
- New serverâ†’agent message: `config_request` â€” push a new RF/FFT/tuner config
  to a running agent.
- New agentâ†’server message: `config_rejected` â€” agent declines a
  `config_request` (e.g. file-based source cannot retune).
- Optional `request_id` field on the agent's `stream_config` and on the
  serverâ†’viewer `stream_config` broadcast: correlates a successful config
  change back to the viewer that initiated it.
- New viewerâ†’server message: `request_config` â€” initiates the chain.
- New serverâ†’viewer message: `request_config_error` â€” reports a per-viewer
  failure (`CONFIG_REJECTED`, `CONFIG_BUSY`, `INVALID_FRAME`, `CONFIG_TIMEOUT`).

All v0.4 messages, fields, and semantics are unchanged.

---

## Transport

- WebSocket only (MVP)
- Auth: `Authorization: Bearer <token>` on HTTP Upgrade (agents); session
  cookie on `/ws/viewer` (browsers)
- Server issues `session_id` in HTTP 101 response header `X-Session-Id`
- Wire encoding negotiated at connect, session-scoped, immutable

## Identity

| Field | Meaning |
|---|---|
| `node_id` | Stable agent install identity. Not hardware serial. Survives hardware swap. |
| `session_id` | Server-issued per connection. Reset on reconnect. |
| `stream_id` | Sub-channel per session. Default `"default"`. Always present. |
| `config_version` | Server-assigned monotonic int per (session, stream_id). In `stream_config_ack` and every frame. |
| `frame_index` | Monotonic per (session, stream_id, config_version). Resets on config change. Gaps = dropped frames. |
| `request_id` | Server-side ID for a single config-change round trip. Generated by the server on receipt of `request_config`, echoed by the agent on its resulting `stream_config` (or `config_rejected`), and broadcast to viewers so the originator can resolve its pending promise. |

```json

  "node_id": "node_a1b2c3",
  "session_id": "ses_01HX...",
  "stream_id": "default",
```

## Control plane â€” server â†’ agent

### connect_ack, stream_config_ack, disconnect, error
Unchanged from v0.4.

### config_request
```json
{
  "msg_type": "config_request",
  "session_id": "ses_01HX...",
  "stream_id": "default",
  "request_id": "req_01HX...",
  "rf": {
    "center_freq_hz": 433920000,
    "sample_rate_hz": 2400000,
    "fft_size": 131072,
    "window_fn": "hann"
  },
  "tuner": {
    "gain_db": 30.5,
    "agc": false
  }
}
```

- `rf` carries the same field set as `stream_config.rf` minus the
  derived/advisory fields (the agent recomputes `baseband_*`, `bin_size_hz`,
  `bin_count`).
- `tuner` is optional. `gain_db: number | null` and `agc: boolean`. When
  `agc=true`, `gain_db` is ignored. Agents with non-tunable sources
  (WAV/SigMF) reject any `config_request` regardless of `tuner` contents.
- `stream_id` matches the agent's current `stream_id` (always `"default"` in
  MVP). Mismatch â†’ agent emits `config_rejected` with `CONFIG_REJECTED`.

Agent behavior on receipt:
1. Validate `session_id` / `stream_id` match.
2. Apply to the source (live retune for capable sources; reject otherwise).
3. Reconfigure the FFT processor.
4. Emit a new `stream_config` carrying the `request_id`.
5. The server assigns the new `config_version` in its `stream_config_ack` and
   fans the new config out to viewers.

If any step fails, the agent emits `config_rejected` instead.

## Control plane â€” viewer â†” server

### viewer â†’ server: subscribe (unchanged)
```json
{ "msg_type": "subscribe", "agent_id": "<uuid>" }
```

### viewer â†’ server: request_config
```json
{
  "msg_type": "request_config",
  "request_id": "req_client_<uuid>",
  "rf": {
    "center_freq_hz": 100000000,
    "sample_rate_hz": 2400000,
    "fft_size": 65536,
    "window_fn": "hann"
  },
  "tuner": { "gain_db": null, "agc": true }
}
```

The viewer-generated `request_id` is for the viewer's own pending-promise
bookkeeping. The server generates its own server-side `request_id` and uses
that for agent-side correlation; the server tells the viewer which server-side
ID corresponds to the viewer's request in either the success path
(`stream_config` broadcast carries the server-side `request_id`, and the
viewer recognizes it by matching against a Map keyed by its own request_id â€”
see "Correlation" below) or the failure path (`request_config_error` carries
both `request_id` fields when applicable).

Validation performed by the server before forwarding:
- `rf.fft_size` is a power of two in `[1024, 131072]`
- `rf.sample_rate_hz âˆˆ {240000, 1024000, 1600000, 2400000, 2560000}` (RTL-SDR
  valid set; the server is the gate even for non-RTL sources)
- `rf.center_freq_hz âˆˆ [1_000_000, 2_700_000_000]`
- `rf.window_fn âˆˆ {"hann"}`
- `tuner.gain_db`, if present, is a number in `[0.0, 49.6]` or `null`

Failure â†’ `request_config_error` with `INVALID_FRAME` to that viewer only.

### server â†’ viewer: stream_config (unchanged shape, optional new field)
```json
{
  "msg_type": "stream_config",
  "agent_id": "<uuid>",
  "session_id": "ses_01HX...",
  "stream_id": "default",
  "config_version": 2,
  "request_id": "req_server_01HX...",
  "rf": { ... },

```

`request_id` echoes the server-side request_id when this stream_config is the

### server â†’ viewer: request_config_error
```json
{
  "msg_type": "request_config_error",
  "request_id": "req_client_<uuid>",
  "code": "CONFIG_REJECTED" | "CONFIG_BUSY" | "INVALID_FRAME" | "CONFIG_TIMEOUT",
  "message": "source does not support live retune"
}
```

`request_id` is the **viewer's** request_id from the originating
`request_config`. Sent only to that viewer's WebSocket.

### Correlation

The viewer keeps a `Map<request_id, {resolve, reject}>`. On success it sees a
`stream_config` whose `config_version` advances past the current one and (when
the server is v0.5+) carries a server-side `request_id`. The simpler rule
used by the v0.5 reference viewer is: once it has sent a `request_config`,
the next `stream_config` it receives is its own (concurrency is serialized
server-side, so no other viewer's change can interleave). The
`request_config_error` path uses an explicit `request_id` match.

---

## Server-side concurrency

Per `(agent session)` only one `request_config` may be in flight. While a
request is pending:
- Server replies `request_config_error` with `CONFIG_BUSY` to any subsequent
  `request_config` from any viewer subscribed to that session.
- The pending entry is cleared on either:
  - Agent's `stream_config` with matching `request_id` (success), or
  - Agent's `config_rejected` with matching `request_id` (failure), or
  - 5 second server-side timeout â†’ `request_config_error` with
    `CONFIG_TIMEOUT` to the originating viewer.

## Errors

Same shape as v0.4. Added codes:

| Code | Fatal | When |
|---|---|---|
| `CONFIG_BUSY` | false | A `request_config` arrived while another is in flight for the same session. Server-only, viewer-facing. |
| `CONFIG_TIMEOUT` | false | Server gave up waiting for the agent to ack a `config_request`. Server-only, viewer-facing. |

`CONFIG_REJECTED` is now also used in the new agentâ†’server `config_rejected`
message (in addition to the prior `stream_config_ack` rejection path).

All other v0.4 codes unchanged.

---

## Module boundaries

Same as v0.4, with one addition: the server now holds a per-session
`pending_config_requests` map and an outbound `agent_send_queue`, allowing it
to push `config_request` to a connected agent and route the agent's response
back to a specific viewer.

The agent's source layer gains an optional `apply_rf_update(rf, tuner)` method
on the `IQSource` protocol. Sources opt in by implementing it.

## Post-v0.5 additions (do not implement now)

- `epoch_ms` (int64) replacing `timestamp_utc` strings
- `data.phase_rad`, `data.psd_db` (requires new `config_version`)
- Per-field encoding config (`uint8`, `float64`)
- `msgpack` wire encoding
- `stream_id` beyond `"default"` (multi-tuner)
- REST snapshot convenience wrapper (`GET /snapshot?node_id=...`)
- Per-source capability advertising (server tells viewer which params are
  tunable for this source so the UI can hide non-applicable controls)
